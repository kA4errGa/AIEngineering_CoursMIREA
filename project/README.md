## 2. Структура проекта

Проект организован в следующей структуре:

- `requirements.txt` – зависимости проекта (библиотеки Python, необходимые для запуска).
- `report.md` – отчёт по проекту (постановка задачи, данные, эксперименты, результаты).
- `self-checklist.md` – чеклист самопроверки проекта перед сдачей.
- `notebooks/` – экспериментальные ноутбуки:
  - EDA, прототипы моделей, отдельные эксперименты.
- `src/` – основной код проекта:
  - модули для работы с данными;
  - обучение моделей;
  - запуск сервиса/скриптов.
- `data/` – демонстрационные/учебные данные:
  - **только открытые, синтетические или обезличенные данные;**
  - при необходимости – небольшие выборки для быстрого запуска.
- `configs/` – конфигурационные файлы:
  - `.yaml`, `.json`, `.env.example` и т.п.
- `tests/` – тесты (юнит-тесты, простые проверки).
- `artifacts/` – сохранённые модели, отчёты, результаты экспериментов.

## 3. Требования и установка

### 3.1. Требования

- Python `3.11.9`.

### 3.2. Установка окружения

```bash
# Перейти в папку проекта
cd project

# Создать виртуальное окружение (опционально, но рекомендуется)
python -m venv .venv

# Активировать окружение:
# Windows:
.venv\Scripts\activate
# Linux / macOS:
source .venv/bin/activate

# Установить зависимости
pip install --upgrade pip
pip install -r requirements.txt
```

## 6. Тесты (если есть)

### Тестирование сервиса

Для проверки работоспособности API были реализованы integration/sanity tests с использованием `pytest` и `FastAPI TestClient`.

Реализованы следующие тесты:

* `test_health` — проверка доступности health-check endpoint `/health`;
* `test_predict` — проверка корректной работы endpoint `/predict` и успешного получения предсказания модели;
* `test_predict_missing_feature` — проверка обработки ошибочного запроса при отсутствии обязательных признаков.

Тесты покрывают:

* корректный запуск FastAPI приложения;
* обработку HTTP-запросов;
* работу ML-модели;
* валидацию входных данных;
* обработку ошибок API.

## 8. Ограничения и дальнейшая работа

 В текущей версии в сервисе реализованая модель имеет не высокое качество предсказаний, нет авторизации и загрузки признаков клиента по id (все признаки передаются в сервис напрямую при запросе).
 В дальнейшем можно добавить:

 - загрузку признаков из базы данных клиентов;
 - увеличить качество модели с помощью большего числа данных;
 - хранение истории запросов пользователей;
 - базовую авторизацию и логирование.
