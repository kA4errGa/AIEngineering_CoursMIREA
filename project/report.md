## 2. Постановка задачи и контекст

1. **Предметную область и зада:**
   - Задача бинарной классификации с табличными признаками.
   - Потенциальный пользователь: работник банка, которому нужно оценить риск невозврата кредита перед его одобрением.
   - Сценарий использования: отправить запрос с признаками клиента и получить предсказание и уверенность в нем.

2. **Формулировку задачи в терминах ML/ИИ:**
   - Входные данные: Данные о заёмщике (31 признак такие как возраст, доход, количество кридитных линий и тд).
   - Целевая переменная: Дефолт.
   - Требования: Результат должен легко интерпретироваться, поэтому помимо самого прогноза модель должна выдавать predict_proba.

3. **Целевые метрики качества:**
   - Метрики качества: целевая метрика F1, помимо неё считаются ROC-AUC, Accuracy, Precision и Recall.
   - Метрика F1 выбрана из-за сильного дисбаланса классов в датасете (типичная проблемма для датасетов оценки кредитного риска).

## 4. Модели и подходы

1. Базовые (baseline) модели

**Использованные модели:**
- Logistic Regression (с L2-регуляризацией, class_weight='balanced')
- Decision Tree (с ограничением глубины, entropy criterion)
- Random Forest (100 деревьев, balanced class_weight)

**Результаты (на валидационной выборке):**

| Модель | Accuracy | F1 | Precision | Recall | ROC-AUC |
|--------|----------|-----|-----------|--------|---------|
| Logistic Regression | 0.674 | 0.331 | 0.217 | 0.693 | 0.749 |
| Decision Tree | 0.657 | 0.289 | 0.190 | 0.601 | 0.683 |
| Random Forest | 0.744 | 0.353 | 0.250 | 0.603 | 0.752 |

**Ключевой вывод:** Random Forest показал наилучший F1 (0.353). Все модели страдают от дисбаланса классов (11.6% дефолтов) — низкий Precision при относительно высоком Recall.

*Ноутбук: `notebooks/baseline.ipynb`*


2. Улучшенные модели и эксперименты

1. **RandomizedSearchCV** для каждой модели (500 итераций для LogReg и DT, 90 для RF)
2. **Расширенные сетки гиперпараметров:**
   - Logistic Regression: C, penalty (l1/l2/elasticnet), tol, max_iter, class_weight
   - Decision Tree: max_depth, min_samples_split, min_samples_leaf, criterion, ccp_alpha, max_features
   - Random Forest: n_estimators, max_depth, min_samples_split, min_samples_leaf, max_features, bootstrap

**Важные настройки:**
- `class_weight='balanced'` — критически важно для работы с дисбалансом
- L1-регуляризация для LogReg (C=0.016, penalty='l1')
- Для Decision Tree оптимальным оказался `max_depth=8, min_samples_leaf=57, ccp_alpha=0.0001`
- Для Random Forest — `bootstrap=False, max_depth=30, n_estimators=127`

**Проведённые эксперименты:**
- SMOTETomek (балансировка классов) — хорошо справлялах на сбалансированных данных, но совершенно не работала на валидационной выборке
- MLP нейросеть с подбором архитектуры и class_weight (F1 до 0.34)

**Результаты улучшенных моделей:**

| Модель | F1 (val) | Прирост к baseline |
|--------|----------|-------------------|
| Logistic Regression | 0.331 | 0.000 |
| Decision Tree | 0.319 | +0.030 |
| Random Forest | 0.365 | +0.012 |

*Ноутбуки: `notebooks/experiments.ipynb`, `notebooks/experimrntXGBoost.ipynb`, `notebooks/experimentBalance.ipynb`*

3. Нейросетевые модели (MLP)

**Архитектуры:**
- архитектура представленна в `configs/experimentMLP_config.json`

**Параметры обучения:**
- `batch_size`: 512
- `learning_rate`: 0.001 (Adam)
- `num_workers`: 0 (Windows совместимость)
- `class_weight` в CrossEntropyLoss — crucial!
- Ранняя остановка (patience=10)
- Gradient clipping (max_norm=1.0)

**Результаты MLP:**
- Без class_weight: F1 ~0.18 (модель не обучалась)
- С class_weight: F1 до 0.34
- SMOTE + MLP: F1 = 0.03 (полный провал, синтетические данные ухудшили обобщение)

**Ключевой вывод:** Для табличных данных с дисбалансом простые модели (Random Forest, XGBoost) дают лучший результат, чем нейросети, без необходимости в сложной настройке.

*Ноутбук: `notebooks/experimentMLP.ipynb`*

## 5. Экспериментальный протокол и результаты

1. Экспериментальный протокол

**Разделение выборки:**
- Данные разделены в соотношении **70% train / 15% val / 15% test**
- Использована **стратификация** по целевому признаку для сохранения пропорции классов (~11.6% дефолтов) во всех выборках
- `random_state=42` для воспроизводимости результатов

**K-Fold валидация:**
- При подборе гиперпараметров использован **StratifiedKFold (n_splits=5)** — стратифицированная кросс-валидация, сохраняющая распределение классов в каждом фолде
- Для Random Forest использован `n_splits=3` (компромисс между скоростью и качеством)

**Расчёт метрик:**
- Метрики считались **на валидационной выборке** в процессе подбора гиперпараметров
- **Финальная оценка** производилась на **тестовой выборке**, не участвовавшей в обучении
- Использованы метрики: Accuracy, F1-score, Precision, Recall, ROC-AUC
- **Основная метрика для оптимизации — F1-score**, так как она лучше отражает качество на несбалансированных данных

2. Сравнение моделей по метрикам (валидационная выборка)

| Модель / конфигурация | Описание | F1 | Recall | Precision | ROC-AUC | Комментарий |
|-----------------------|----------|-----|--------|-----------|---------|-------------|
| **Baseline Logistic Regression** | L2-регуляризация, balanced class_weight | 0.331 | 0.693 | 0.217 | 0.749 | Высокий recall, но очень низкая precision |
| **Baseline Decision Tree** | max_depth=15, min_samples_split=100, entropy | 0.289 | 0.601 | 0.190 | 0.683 | Слабая обобщающая способность |
| **Baseline Random Forest** | 100 деревьев, balanced class_weight | 0.353 | 0.603 | 0.250 | 0.752 | Лучший среди baseline |
| **Optimized Random Forest** | bootstrap=False, max_depth=30, n_estimators=127 | **0.365** | 0.459 | 0.303 | 0.752 | **Финальная модель** |
| **Optimized Decision Tree** | max_depth=8, min_samples_leaf=57, ccp_alpha=0.0001 | 0.319 | 0.635 | 0.213 | 0.726 | Улучшение на +0.03 к baseline |
| **MLP + Class Weight** | 3 слоя, BatchNorm, Dropout(0.3), class_weight в loss | 0.340 | 0.458 | 0.302 | 0.750 | Сравним с RF, но сложнее |
| **XGBoost (по умолчанию)** | scale_pos_weight=7.6 | 0.350 | — | — | — | Результат без перебора параметров |

**Ключевые наблюдения:**
- Logistic Regression даёт самый высокий Recall (0.693), но низкий Precision (0.217) — модель слишком агрессивно предсказывает дефолты
- Random Forest показывает лучший баланс между Precision и Recall
- SMOTETomek полностью провалился (F1=0.03) — синтетические данные не обобщаются
- MLP после добавления `class_weight` показал результат, сравнимый с Random Forest

3. Выбор финальной модели

**Финальная модель:** `RandomForestClassifier` с параметрами:

```python
RandomForestClassifier(
   random_state=42,
   class_weight="balanced",
   bootstrap=False,
   criterion="gini",
   max_depth=30,
   max_features="sqrt",
   min_samples_leaf=10,
   min_samples_split=37,
   n_estimators=127
)
```

**Результаты на тестовой выборке:**
- **F1-score: 0.368**
- Recall: 0.466
- Precision: 0.304
- ROC-AUC: 0.755

**Причины выбора Random Forest:**
1. **Лучший F1 среди всех моделей** (0.365 против 0.353 у baseline)
2. **Лучший баланс Precision/Recall** — модель не слишком агрессивна, но и не слишком консервативна
3. **Стабильность** — меньше подвержена переобучению, чем Decision Tree
4. **Интерпретируемость** — возможность анализа важности признаков
5. **Простота инференса** — не требует сложной предобработки (в отличие от нейросетей)
6. **Быстрый инференс** — важен для сервиса реального времени

**Trade-off’ы:**
| Фактор | Решение | Обоснование |
|--------|---------|-------------|
| Точность vs Скорость | Выбрана точность | Сервис не требует миллисекундных задержек |
| Сложность vs Простота | Выбрана простота | Random Forest легче деплоить и поддерживать |
| Recall vs Precision | Компромисс | F1 как основная метрика обеспечивает баланс |
| Переобучение vs Обобщение | Умеренная регуляризация | min_samples_leaf=10, max_depth=30 |

**Почему не выбраны другие модели:**
- **XGBoost** — показал схожий результат (0.35), но требует больше памяти и сложнее в деплое
- **MLP** — сложнее в инференсе, требует масштабирования признаков, результат не лучше Random Forest
- **SMOTE** — полностью провалился

## 6. Архитектура решения и сервис

1. Архитектура пайплайна

Проект реализует полный ML pipeline для оценки кредитного риска клиентов.
Общий процесс работы системы:

```text
Пользователь -> FastAPI сервис -> Предобработка входных данных -> ML-модель -> Предсказание -> JSON-ответ
```

Этапы работы:

1. Пользователь отправляет HTTP-запрос на endpoint `/predict`;
2. FastAPI получает и валидирует входные данные;
3. Данные преобразуются в `pandas.DataFrame` с сохранением правильного порядка признаков;
4. Загружается обученная модель (`joblib`);
5. Модель выполняет предсказание;
6. Сервис возвращает результат в формате JSON.

Модель и список признаков сохраняются в виде artifacts:

```python
{
    "model": model,
    "feature_columns": feature_columns
}
```

Это позволяет гарантировать корректный порядок признаков при inference.

---

2. API и endpoints

В сервисе реализованы следующие endpoints:

### `GET /health`

Проверка доступности сервиса.

Ответ:

```json
{
  "status": "ok"
}
```

---

### `GET /model-info`

Возвращает информацию о модели и список признаков.

---

### `POST /predict`

Основной endpoint для выполнения предсказания кредитного риска.
Endpoint принимает JSON с признаками клиента:

```json
{
  "features": {
    "Age": 56,
    "Income": 85994
    ...
  }
}
```

В ответ сервис возвращает:

```json
{
  "prediction": 1,
  "probability": 0.93
}
```

Также реализована обработка ошибок при отсутствии обязательных признаков.

3. Технологический стек

В проекте используются следующие технологии:

* `Python 3.11`
* `FastAPI` — реализация REST API;
* `Uvicorn` — ASGI сервер;
* `Pydantic` — валидация данных;
* `Pandas` и `NumPy` — обработка данных;
* `Scikit-learn` / `XGBoost` — ML модель;
* `Pytest` — тестирование API;
* `Joblib` — сериализация модели;
* `Docker` — контейнеризация сервиса.

4. Docker

Сервис запускается в Docker-контейнере.

Команды для запуска:
